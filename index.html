git指南

集中式（svn）

  • 对比：因为每次存的是差异，需要的硬盘空间会相对的小一点，可是回滚速度会很慢

  • 优点：代码存放在单一的服务器上， 便于项目的管理

  • 缺点：服务器宕机：员工写的代码等不到保障

    ​ 服务器炸了：整个项目的历史记录都会丢失

分布式（git）

  • 对比：git每次存的是项目的完整快照，需要的硬盘空间会相对的大一点（团队对代码做了极致的压缩 最终需要的实际空间比svn躲不了太多，可是Git的回滚速度极快）

  • 优点：完全的分布式

  • 缺点：学习起来比svn陡峭

  1. 原理图（创建模型）
    工作区域
    • 工作区：我们平时写代码的地方，此时git不会管理我们的代码
    • 暂存区：放数据对象的地方
    • 版本库：最终的代码实现提交到这里 .git目录就是版本库
    • 远程仓库
    git对象

    • 数据对象：一个文件代表一个数据对象

      • 结构：key:value(hash/value)
      • 作用：存储文件内容
      • 缺点：记住文件的每一个版本对应的hash值并不现实
      • 缺点：文件名并没有被保存，我们仅保存了文件的内容

    • 树对象

      • 作用：解决文件名保存的问题，允许我们将多个文件组织到一起
      • 结构：hash：指针
      • 缺点：不知道hash值对应的是哪一个版本
      • 缺点：不知道这个版本的一些基本信息

    • 提交对象: 包含一个树对象条目，代表着当前项目快照，从这树对象开始我们就能找到所有提交的数据对象，从而形成Git中的一个版本

      • 作用1：给树对象做一层封装（包含项目的基本信息）解决了树对象存在的问题
      • 作用2：提交对象前有一个活动指针HEAD，指向master分支，当有最新的提交对象时，HEAD会携带着master分支往前走

    文件状态
    • 未跟踪/已跟踪
      • 已暂存/已修改/已提交

  2. 常用命令

    • git操作最基本的流程

  • Git高级命令（分支）

  • Git团队协作

  1. 时间复杂度(对一个算法运行时间长短的量度)

    • 基本操作执行次数（ T(n) = 3n, T(n) = 5logn, T(n) = 2, T(n) = 0.5n 2 + 0.5n ）

    • 渐进时间复杂度

      • 如何推导出时间复杂度
        • 如果运行时间是常数量级，则用常数1表示
        • 只保留时间函数中的最高阶项
        • 如果最高阶项存在，则省去最高阶项前面的系数

      T(n) = 3n => T(n)=O(n)

      T(n) = 5logn => T(n) =O(logn)

      T(n) = 2 => T(n) =O(1)

      T(n) = 0.5n 2 + 0.5n => T(n) =O(n 2 )

计算机基础

  1. 电脑硬件

    • 内存：读写速度快
    • 硬盘：读写速度慢

  2. 内存模拟

    • 一个小格只能存储0或1（单位：bit）8个小格等于一个字节（单位：byte）
    • 内存：只负责数据的存放，不负责数据写入，断电就会全部清空
    • 硬盘：负责数据的存放，还有写入，断电不会清空

  3. 传输过程

    • 例子：450 * 950= ？
    • 地址总线：告诉内存读/写的位置
    • 数据总线：数据传输
    • 控制总线：告诉内存是要读/写
    • 总线：一根线只能传输1/0（2^n种情况）

  4. CPU

    • CPU擅长整数运算，GPU擅长小数运算
    • 32位CPU: 32根线 （2^32=4GB）
    • 64位CPU: 64根线 （2^64=8GB）

    • 1，2，4，8 字节便于寻址

electron

与web开发的不同点

  1. 主进程和渲染进程

    • 主进程：Electron运行package.json的main脚本的进程

      • 特点：每个应用只有一个主进程

      • 作用：管理原生GUI, 典型的窗口（BrowserWindow, Tray, Dock, Menu）

        ​ 创建渲染进程

        ​ 控制应用生命周期（app）

    • 渲染进程：展示Web页面的进程

      • 特点：一个Electron应用可以有多个渲染进程

      • 作用：通过Node.js，Electron提供的API可以跟系统底层打交道

        ​ 开发Web页面

  2. 进程间的通信

    • 目的：通知事件（比如主进程才能够创建原生菜单）

      ​ 数据传输（比如想在某个页面里获得现在的内存情况）

      ​ 共享数据（比如用户信息在各个进程中都需要用到）

    • 类型：

      • 从渲染进程到主进程
        • Callback写法
          • ipcRenderer.send(channel, …args)
          • ipcMain.on(channel, handler)
        • Promise写法（Electron7.0之后，处理请求+响应模式）
          • ipcRenderer.invoke(channel, …args)
          • ipcMain.handle(channel, handler)
      • 从主进程到渲染进程
        • webContents.send(channel)
        • ipcRenderer.on(channel, handler)
      • 渲染进程到渲染进程（比较少）
        • 通知事件
          • 通过主进程转发（Electron之前）
          • ipcRenderer.sendTo（Electron5之后）
        • 数据共享
          • Web技术
          • 使用remote（尽量少用）

  1. Native能力及原生GUI

  2. 更好的体验

    • 无兼容性问题
      • 不用担心在Safari, IE上的表现差异了
      • 大胆使用Chrome浏览器已经支持的API
      • babel中设置targets为Electron对应的Chrome版本

  • 大胆使用ES6/7/8/9/10高级语法

    • Async await/Promise
    • String/Array/Object等高级用法
    • BigInt

  • 无跨域问题

    • 使用Node.js发送请求
    • 使用Electron net发送请求

  • 最新浏览器Feature

  • No PolyFill

  • Powered by Node.js

  • More

    • 操作本地文件
    • 更好用的本地DB
    • 多线程，多进程并行
